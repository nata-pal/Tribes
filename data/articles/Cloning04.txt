class="date-posts">          <div class='post-outer'> <div class='post hentry'> <a name='4898225104883479355'></a> <h3 class='post-title entry-title'> <a href='http://blog.cloudbees.com/2015/08/jenkins-workflow-using-global-library_7.html'>Jenkins Workflow - Using the Global Library to implement a re-usable function to call a secured HTTP Endpoint</a> </h3> <div class='post-header'> <div class='post-header-line-1'></div> </div> <div class='post-body entry-content'> <div class="p1"><span class="s1">Jenkins</span> <span class="s1">Workflow</span> - Using the Global Library to implement a re-usable function to call a secured HTTP <span class="s1">endpoint</span></div><div class="p2"><br /> </div><div class="p1">This <span class="s1">blog</span> post will demonstrate how to access an external system using HTTP with that is protected by a session based logon. This type of integration is often required to retrieve information to be used in the <span class="s1">workflow</span>, or, to trigger events on existing external systems, such as a deployment framework. These types of systems often have some form of authentication mechanism - it may be Basic <span class="s1">Auth</span> based or, using some form of session-based security. This <span class="s1">blog</span> post will show how you can use the following features</div><div class="p1"></div><ul><li><span class="s1">Jenkins</span> <span class="s1">Workflow</span> <span class="s1">plugin</span></li> <li><span class="s1">Git</span>-based <span class="s1">Workflow</span> Global Library repository</li> <li>curl cookie handling</li> </ul><br /> <h1><span style="font-size: x-large;">Getting Started</span></h1><h3>Requirements</h3><div class="p1"></div><ol><li>JDK 1.7+</li> <li><span class="s1">Jenkins</span> 1.609+</li> <li>An external system secured with session based authentication - (in this example I use a <span class="s1">Jenkins</span> server with security enabled as an example - although it is not recommended you do this for real as <span class="s1">Jenkins</span> has better token based APIs to use)</li> </ol><div class="p1"><h3>Installation</h3></div><div class="p1"></div><ol><li><a href="http://www.oracle.com/technetwork/java/javase/downloads/index.html">Download and install </a>JDK 1.7 or higher</li> <li><a href="https://jenkins-ci.org/">Download</a> and <a href="https://wiki.jenkins-ci.org/display/JENKINS/Installing+Jenkins">install</a> <span class="s1">Jenkins</span></li> <li><a href="https://wiki.jenkins-ci.org/display/JENKINS/Starting+and+Accessing+Jenkins">Start <span class="s1">Jenkins</span></a></li> </ol><div class="p1"><h3>Setup <span class="s1">Jenkins</span></h3></div><div class="p1"></div><ul><li><b>Update <span class="s1">plugins</span> </b>- Make sure you have the latest <span class="s1">Workflow</span> <span class="s1">plugins</span> by going to Manage <span class="s1">Jenkins</span> &#8211;&gt; Manage <span class="s1">Plugins</span> -&gt; Updates and selecting any <span class="s1">Workflow</span>-related updates. Restart <span class="s1">Jenkins</span> after the updates are complete. As of this writing the latest version is 1.8</li> <li><b>Global libraries <span class="s1">repo</span> </b>- <span class="s1">Jenkins</span> exposes a <span class="s1">Git</span> repository for hosting global libraries meant to be reused across multiple CD pipelines managed on the Master. We will setup this repository so you can build on it to create your own custom libraries. If this is a fresh <span class="s1">Jenkins</span> Install and you haven&#8217;t setup this <span class="s1">git</span> repository, follow these instructions to setup.</li> </ul><div class="p2"><br /> </div><div class="p1"><span style="color: red;"><b><i>Important - Before proceeding to the next steps, make sure your <span class="s1">Jenkins</span> instance is running</i></b></span></div><div class="p2"><br /> <br /> </div><div class="p1">See <a href="https://github.com/jenkinsci/workflow-plugin/blob/master/cps-global-lib/README.md"><span class="s1">Workflow</span> Global Library</a> for details on how to set up the shared library - note that if security is enabled, the <span class="s1">ssh</span> format works best.</div><div class="p2"><br /> </div><div class="p1">If using <span class="s1"><b><i>ssh</i></b></span> ensure that your <a href="https://wiki.jenkins-ci.org/display/JENKINS/Jenkins+CLI">public key has been configured</a> in <span class="s1">Jenkins</span>.</div><div class="p2"><br /> </div><div class="p1">To initialise the git repository:</div><div class="p2"><br /> </div><div class="p1"><pre class="brush: bash">git clone ssh://&lt;USER&gt;@&lt;JENKINS_HOST&gt;:&lt;SSH_PORT&gt;/workflowLibs.git</pre></div><div class="p2"><br /> </div><div class="p1">Where</div><div class="p1"></div><ul><li>USER is a user a valid user that can authenticate&nbsp;</li> <li>JENKINS_HOST is the DNS name of the <span class="s1">Jenkins</span> server you will be running the <span class="s1">workflow</span> on. If running on the CloudBees <span class="s1">Jenkins</span> platform, this is the relevant Client Master not the <span class="s1">Jenkins</span> Operations <span class="s1">Center</span> node.</li> <li>SSH_PORT is the <span class="s1">ssh</span> port defined in <span class="s1">Jenkins</span> configuration:</li> </ul><div class="separator" style="clear: both; text-align: center;"><a href="http://3.bp.blogspot.com/-yrYeIj3elmA/VcT2nibCD_I/AAAAAAAAAVU/6ixOWuSuGhk/s1600/Screen%2BShot%2B2015-08-05%2Bat%2B12.15.33.png" imageanchor="1" style="clear: left; float: left; margin-bottom: 1em; margin-right: 1em;"><img border="0" height="56" src="http://3.bp.blogspot.com/-yrYeIj3elmA/VcT2nibCD_I/AAAAAAAAAVU/6ixOWuSuGhk/s640/Screen%2BShot%2B2015-08-05%2Bat%2B12.15.33.png" width="640" /></a></div><br /> <br /> <div class="p1">Note the repository is initially empty.</div><div class="p2"><br /> </div><div class="p1">To set things up after cloning, start with:</div><div class="p1"><pre class="brush: bash">git checkout -b master</pre></div><div class="p1">Now you may add and commit files normally. For your first push to <span class="s1">Jenkins</span> you will need to set up a tracking branch:</div><div class="p1"><pre class="brush: bash">git push --set-upstream origin master</pre></div><div class="p1">Thereafter it should suffice to run:</div><div class="p1"><pre class="brush: bash">git push</pre></div><div class="p1"><h1><span style="font-size: x-large;">Creating a Shared Function to Access Jenkins</span></h1><br /> </div>Create the groovy script in the <span class="s1">workflow</span> library:<br /> <div class="p2"><pre class="brush: bash">cd workflowLibs mkdir -p src/net/harniman/workflow/jenkins curl -O \ https://gist.githubusercontent.com/harniman/8f1418af794d26035171/raw/941c86041adf3e9c9bfcffaf9e650e3365b9ee55/Client.groovy git add * git commit git push </pre></div><div class="p1">This will make the following script</div><div class="p2"></div><div class="p1"><pre class="brush: groovy">&nbsp; &nbsp; net.harniman.workflow.jenkins.Client</pre></div><div class="p2"></div><div class="p1">available for use by <span class="s1">workflow</span> scripts using this syntax:</div><div class="p2"></div><div class="p1"><pre class="brush: groovy">&nbsp; &nbsp; def jc = new net.harniman.workflow.jenkins.Client()</pre></div><div class="p2"></div><div class="p1">and methods accessed using:</div><div class="p2"></div><div class="p1"><pre class="brush: groovy">&nbsp; &nbsp; &nbsp;def response=jc.test(&lt;host&gt;, &lt;user&gt;, &lt;pass&gt;, &lt;cmd&gt;)</pre></div><div class="p2"><br /> </div><div class="p1">Note:</div><div class="p1"></div><ul><li>We must NOT define an enclosing class for the script as we want to call step functions such as <span class="s1"><b><i>sh</i></b></span>&nbsp;- see <a href="https://github.com/jenkinsci/workflow-plugin/tree/master/cps-global-lib">https://github.com/jenkinsci/workflow-plugin/tree/master/cps-global-lib</a> for further details on this restriction.</li> <li>scripts follow the usual <span class="s1">Groovy</span> package naming formats and thus need to be in the appropriate directory structure</li> <li>it is unnecessary to go out of process to access this <span class="s1">Jenkins</span> instance - it can be accessed from <span class="s1">Groovy</span> via the <span class="s1">Jenkins</span> model, however, this is to show how a client for form based access could be built</li> <li>The below script is built to access <span class="s1">Jenkins</span> as an example for demonstration</li> </ul><br /> <div class="p1">This is the actual contents of Client.groovy:</div><div class="p2"><br /> </div><pre class="brush: groovy">package net.harniman.workflow.jenkins String test (host, user, pass, <span class="s1">cmd</span>) { &nbsp; &nbsp; node("shared") { &nbsp; &nbsp; &nbsp; &nbsp; init="curl -s -c .cookies ${host}" &nbsp; &nbsp; &nbsp; &nbsp; userAttr="j_username=${user}" &nbsp; &nbsp; &nbsp; &nbsp; passAttr="j_password=${pass}" &nbsp; &nbsp; jsonAttr="json=%7B%22j_username%22%3A+%22${user}%22%2C+%22j_password%22%3A+%22${pass}%22%2C+%22remember_me%22%3A+false%2C+%22from%22%3A+%22%2F%22%7D" &nbsp; &nbsp; &nbsp; &nbsp; login="curl -i -s -b .cookies -c .cookies -d $userAttr -d $passAttr -d 'from=%2F' -d $jsonAttr -d 'Submit=log+in' $host/j_acegi_security_check" &nbsp; &nbsp; &nbsp; &nbsp; cmd="curl -L -s -b .cookies -c .cookies $host/$cmd" &nbsp; &nbsp; &nbsp; &nbsp; echo "Initialising HTTP Connection" &nbsp; &nbsp; &nbsp; &nbsp; sh "${init} &gt; .init 2&gt;&amp;1"&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; echo "Performing Login" &nbsp; &nbsp; &nbsp; &nbsp; sh "${login} &gt; .login 2&gt;&amp;1" &nbsp; &nbsp; &nbsp; &nbsp;&nbsp;def loginresponse = readFile '.login' &nbsp; &nbsp; &nbsp; &nbsp; if (loginresponse =~ /Location:.*loginError/) { &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; echo "Error loging in" &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; error"Unable to login. Response = $loginresponse" &nbsp; &nbsp; &nbsp; &nbsp; } &nbsp; &nbsp; &nbsp; &nbsp; echo "Invoking command" &nbsp; &nbsp; &nbsp; &nbsp; sh "${cmd} &gt; .output 2&gt;&amp;1" &nbsp; &nbsp; &nbsp; &nbsp;&nbsp;def output = readFile '.output' &nbsp; &nbsp; &nbsp; &nbsp; sh "rm .init .login .output" &nbsp; &nbsp; &nbsp; &nbsp; return output &nbsp; &nbsp; } } </pre><div class="p2"><br /> </div><div class="p1"><div class="p1"><h1>Creating the workflow</h1></div><div class="p1"></div>Create a new workflow job with the Workflow Definition as follows:</div><div class="p2"><br /> </div><pre class="brush: groovy">def jc = new net.harniman.workflow.jenkins.Client() def response=jc.test("http://localhost:8080&#8221;, "annie.admin", "password", "whoAmI") echo "=======================" echo response </pre><div class="p2"></div><div class="p1"><br /> Ensure you substitute in real values for:</div><div class="p1"><ul><li>&lt;host&gt;</li> <li>&lt;user&gt;</li> <li>&lt;pass&gt;</li> <li>&lt;cmd&gt;</li> </ul></div><div class="p2"></div><div class="p1">You can substitute in the required <b><i>cmd</i></b> for the query - for instance, it could be &lt;jobname&gt;/config.xml to retrieve a job&#8217;s config.</div><div class="p2"><br /> </div><div class="p1"><div class="p1"><h1>Run the Job</h1></div><div class="p1"></div>Time to check that it all works, so go ahead and trigger the build.&nbsp;</div><div class="p2"><br /> </div><div class="p1">If it works successfully you should see the raw body of the response printed in the console log.</div><div class="p2"><br /> </div><div class="p1"><div class="p1"><h1>Further Improvements</h1></div><div class="p1"></div>If you look closely you can see in this example, the username and password is output to the console. This is not recommended, and you can avoid this by integrating with credentials and having the workflow pass in the credentials ID to be used. See <a href="http://developer-blog.cloudbees.com/2015/01/workflow-integration-for-credentials.html">this article</a> for more details on the syntax with a <b><i>sh</i></b> step using <b><i>curl</i></b>.</div><div class="p2"><br /> </div><div class="p1"><h1>Conclusion</h1></div><div class="p1">The Workflow Global Library provides the ability to share common libraries (groovy scripts) across many workflow jobs to help keep workflows <a href="https://en.wikipedia.org/wiki/Don%27t_repeat_yourself">DRY</a>.</div><div class="p2"><br /> </div><div class="p1">In the original version of this post, we used the Groovy built in HTTP libraries to make the HTTP call. On further investigation and testing of failure scenarios we found this to be brittle when the master restarts, The underlying reason for this behaviour is that ALL&nbsp; the workflow logic runs within the process space of the master, this includes any logic operating within a <b><i>node(){}</i></b> block.<br /> <br /> To ensure survivability of the workflow that is executing during a master restart, it is necessary to perform the bulk of the work inside <b>workflow steps</b>. These are designed to be passed to a separate executor for execution rather than executed within the master&#8217;s process space. We have therefore moved the HTTP call, which may take an amount of time to run, to be called within a <b><i>sh</i></b> step using <b><i>curl</i></b>.<br /> <br /> We have still enabled the packaging of this into a shared library module, thus simplifying the workflow script and promoting re-use. It is necessary to ensure subsequent <b><i>curl</i></b> requests share the cookiejar. Should you want to leverage the power of groovy to perform such a call, this should be done by calling a separate groovy script from an <b><i>sh</i></b> or <b><i>bat</i></b> step.</div><div
